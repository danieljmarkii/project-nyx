Round 2's spine NEW

The widget only one-press-logs what it can name. Otherwise it's a fast door, never a guesser.
  • Meal slots are bound to a named food. During a diet trial, "Dinner" is the trial diet by definition; outside one, it's the slot's learned usual (confirmation-over-entry — the app already knows it after week one). Tapping the slot logs that food, assumed portion, no intake rating. If no stable usual exists for a slot, the tap deep-links into quick-log instead of logging a guess.
  • The treat button never logs "a treat." It flips the widget in place (iOS 17 interactive state) to the pet's 2–3 most-logged named treats + "Something else…" (deep-link). Two taps, real item, real exposure record.
  • No generic events exist in the schema path. There is nothing to "edit later" because an unnameable event never gets written — the failure mode the PM called out is structurally impossible, not policy-discouraged.
Data ScientistThis is the same shape as the meal-confirmation card: the cheap path is also the honest path. A slot-bound log carries logged_via='widget' + the named food_item_id — fully usable by the correlation engine.
JordanTwo taps for a named treat is still faster than unlock-find-launch, and I actually know which treat I just gave. Fine trade.

Candidate D DEFERRED — PHASE 2

Lock Screen accessories

Per the PM: lower priority than the Home Screen widget itself. Parked with its round-1 design + the T&S pre-auth rule (slot state + counts only; never symptom text or AI language). Cheap to add once the widget target exists; sequenced behind the flagship in the PR plan.

Held constant across every candidate

  • Logs only what it can name (round 2's spine) — no generic events; the unnameable path deep-links, never guesses.
  • Fail-safe honesty (B-156 G1 generalized): no surface ever assumes an event into existence. An unconfirmed slot is a visible gap, not a ✓.
  • Provenance: every widget write carries logged_via='widget' + the named item id (B-289 lands first — unbackfillable later).
  • Per-widget pet binding: configured at placement; never follows the in-app active-pet switch (B-086).
  • Never reassures: no "all clear," no Signal/AI-read copy in v1 (stale-refresh hazard — deliberately deferred).
  • No monetization state (D5 parity with Home) — and the widget itself ships free (PM ruled 2026-07-24: easier logging → better retention → more insight; amends the D-M1 "Home-screen widgets" Premium bullet, doc edits flagged).
  • Silent by design: zero notifications, Principle 4 untouched. One accent: teal on interactive elements only.
