# Culprit — Pre-Submission Hardening Audit (n=1 → real users)

**Created:** 2026-07-20 · **Owner:** full product team (PM-requested persona sweep)
**Companions:** [`docs/app-store-readiness.md`](./app-store-readiness.md) (what gates submission) · [`docs/app-store-submission-guide.md`](./app-store-submission-guide.md) (how/in what order). This doc is the third lens: **what hardening the app needs before strangers use it** — the persona-by-persona status the PM asked for, grounded in four independent review passes (App-Store compliance sweep · `rls-privacy-reviewer` multi-tenant attack pass · Dir.-of-Eng hardening audit · first-run product/Jordan-Sam walk) plus live production checks (Supabase advisors, bucket postures, deployed-function inventory, migration history, `npm audit`).

## Bottom line

The app is in **much better submission shape than the trackers say** — the legal/deletion/permission-string/version-display cluster that dominated the readiness register is *done and verified in code*. What the sweep actually found is different from what was being tracked:

1. **One real multi-tenant boundary hole:** `nyx-vet-attachments` is readable **bucket-wide by any authenticated user** (B-248, known but parked at `Later`). Dormant at n=1; a live cross-tenant read of vet documents (prescriptions, discharge summaries) the moment user #2 uploads one. One migration, mirroring the already-shipped 025.
2. **One unaudited public path live in prod:** a `view-report` Edge Function is **deployed with `verify_jwt=false`** and has **no source in the repo** — it's the PR-6 public-share route, live early, without the mandated `rls-privacy-reviewer` gate. Nothing in the app links to it → delete it today, rebuild it properly at PR 6.
3. **The three genuine submission gaps are small:** the `paywall_enabled=false` flip (the mock still shows a £-priced trial CTA), a **forgot-password path** (none exists — B-280), and the **App Review demo account** (B-271, spec'd, not built).
4. **The Dir.-of-Eng no-brainers are automation, not code quality:** CI (0 automated checks on ~400 merges, while ~2,270 tests pass locally), crash reporting + an error boundary (a production crash today is a silent white screen), and a sync poison-pill fix (one bad row can silently wedge a table's queue forever).

Everything else the personas own — RLS table coverage, service-role confused-deputy boundaries, AI cost caps, session/auth storage, empty states, voice, clinical never-reassure — **held under attack**. The detailed sign-offs are below.

## C. The Dir. of Engineering's no-brainers (cheap, high-leverage, pre-scale)

Ranked. 1–3 are the ones I'd stop the line for.

1. **Ship CI** (B-390, filed at the retro — this audit seconds it as the top item). `.github/workflows/` does not exist; the only gate is the opt-in local pre-push hook. ~1,463 Jest cases + ~809 Deno cases already pass — the work is one workflow file: `npm ci && tsc --noEmit && jest` (self-contained today) + a `setup-deno` job for the function suites (needs network for remote imports). Turns "400 merges, 0 checks" into an enforced gate before real users arrive.
2. **Crash reporting + root ErrorBoundary** (B-016). Nothing is wired — an unhandled production error is a white screen with zero telemetry, and an App-Review "it crashed" is untriageable. Sentry (RN + Edge Functions) + an ErrorBoundary around the root Stack + `ErrorUtils.setGlobalHandler`. This also resolves the B-047/B-016 "unowned gate" PM action already in STATUS.
3. **Sync poison-pill isolation** (B-397, new). Batch tables (`meals`, `events`, `weight_checks`, `feeding_arrangements`, `medications`) upsert up to 100 rows and on any error mark **nothing** synced (`lib/sync.ts:202-209,328`) — one structurally-bad row wedges that table's queue forever, with no retry counter, no backoff, and **invisibly**: the pending badge counts only `events` (`lib/db.ts:950-956`). Fix = the per-row `continue` pattern already used for attachments + `sync_attempts` quarantine + widen `getSyncStatus()`.
4. **Automated deploy-drift check** (B-178) — made vivid this session by finding drift in the *other* direction (a deployed function with no repo source). A scheduled job diffing each deployed function's sha256 against the repo bundle closes both directions.
5. **`npm audit fix` + Expo patch bump.** 1 critical + 3 high (shell-quote, ws, undici, form-data) — all in the Expo/RN toolchain, not app runtime code; cheap to clear and store-hygienic.
6. **Supabase config hardening (B-402, new — all dashboard/SQL, ~15 min):** enable leaked-password protection (advisor WARN); revoke `anon`/`authenticated` EXECUTE on `handle_new_user` (RPC-exposed SECURITY DEFINER — likely inert but wrong-shaped); assess `record_ai_usage` REST exposure (self-increment only — harmless, but revoke REST EXECUTE if unneeded); later: `(select auth.uid())` initplan rewrite migration (perf-at-scale WARN on ~all policies), unindexed-FK indexes, percentage auth pool.
7. **Bound `generate-signal` spend per-user** (B-399, new). Its cap is per-**pet** (12/day, 240/mo) and pet creation is uncapped, so N pets → N×cap Anthropic spend; every other AI function is per-user-bounded. Add a per-user aggregate cap (or cap pets-per-account server-side — composes with the ratified 4+-pet Premium gate, T3-F). Also note: cap enforcement is deliberately fail-open on RPC error — fine, but it's the thing to watch once B-016 telemetry exists.

None is large; all become 10× harder to retrofit after launch. `tests: 1,463 jest + 809 deno green locally; enforced nowhere` — that sentence is the whole case for item 1.

**Trust & Safety / Privacy** — *Boundaries: strong core, two edges.* The attack pass broke nothing in the table-level RLS or any service-role function — genuinely well-built, and deletion held end-to-end.
