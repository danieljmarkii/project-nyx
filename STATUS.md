**Cutting a new TestFlight build (verified 2026-06-12 — build 22 from `main`).** TestFlight binaries are built with the **`production` profile** (`distribution: store`, `channel: production`, `autoIncrement: true` — builds 7, 8, 22 all used it). One command builds the store-signed binary and submits that *same* artifact:
```
eas build --platform ios --profile production --auto-submit
```
**Two traps that cost the 2026-06-12 session — do not repeat:**
- **Never build TestFlight with the `preview` profile.** `preview` is `distribution: internal` (ad-hoc signed for direct device install) — Apple/TestFlight only accept **store-signed** builds, so an internal binary is never submittable. We built 10/11/21 on `preview` and none could be uploaded.
- **Never submit with `eas submit --latest` for this.** `--latest` resolves to the latest *store* build and silently skips internal ones, so it re-uploaded a **stale build 8** every time while our new internal builds sat ineligible — the "build number 8 already used" error never moved no matter how high we cranked the counter. `--auto-submit` binds the upload to the binary just built, which kills the ambiguity.
- Build numbers live on EAS (`appVersionSource: remote`); `autoIncrement` advances them. A "build number N already used" rejection means EAS's counter is *behind* App Store Connect — push it past ASC's max with `eas build:version:set --platform ios`, then rebuild.

**OTA to the installed TestFlight build = `eas update --branch production`** — the build's channel is `production`, NOT `preview`. (Corrected 2026-06-12; the prior "`eas update --branch preview` reaches it" note here was wrong — `preview`-channel updates never reach a `production`-channel build, and that error is what steered the session down the preview path.)
